1. Стратегия AOT Generation 
Вместо вычисления дат при каждом GET-запросе, система генерирует конкретные экземпляры задач в таблицу tasks на перед.
Почему: Это обеспечивает мгновенную скорость работы (O(1)) при получении списка задач на день, так как база использует простой индекс по due_date.
Профит: Врачи могут видеть своё расписание на месяц вперед в стандартном календаре.
2. Транзакции
Создание шаблона и генерация серии задач обернуты в единую транзакцию (pgx.Tx).
Почему: Недопустима ситуация, когда шаблон создан, а задачи не нарезались (или наоборот). Либо создается всё, либо ничего.
3. Умная синхронизация при обновлении 
При изменении настроек периодичности в шаблоне, система:
Удаляет все будущие невыполненные задачи.
Пересчитывает расписание и создает новые задачи.
Если изменился только заголовок — обновляет метаданные в существующих задачах без перегенерации.
4. Фоновый воркер
Реализована горутина-воркер, которая раз в сутки проверяет активные шаблоны.
Логика: Если до конца сгенерированного периода осталось менее 25 дней, воркер автоматически «докидывает» задачи до конца следующего месяца.

 Обоснование проектных решений и допущения
1. Жизненный цикл задачи 
Решение: Разделение на таблицы task_templates (правила) и tasks (экземпляры).
Обоснование: В медицинской практике важно иметь возможность отредактировать конкретную задачу (например, добавить фамилию пациента в обход на вторник), не меняя общее правило для всей серии.
2. Ограничение горизонта планирования
Предположение: Система генерирует задачи только на один календарный месяц вперед.
Обоснование: Бесконечная генерация избыточна и перегружает БД. Месячный интервал достаточен для планирования работы персонала и легко поддерживается фоновым воркером.
3. Обработка «коротких» месяцев
Предположение: Если задача установлена на 31-е число, а в месяце 28 или 30 дней, задача создается в последний доступный день этого месяца.
Обоснование: Медицинские процедуры (например, ежемесячная отчетность) должны выполняться в рамках отчетного периода, а не пропускаться из-за календарных особенностей.
4. Синхронизация при изменениях 
Решение: При изменении типа периодичности (например, с Daily на Monthly) все невыполненные будущие задачи старой серии удаляются.
Обоснование: Это предотвращает «замусоривание» календаря врача неактуальными задачами при смене рабочего графика.
5. Soft Deactivation
Решение: При удалении шаблона я использую флаг is_active = false.
Обоснование: В медицинских ИС критически важно сохранять историю (аудит). Мы прекращаем генерацию новых задач, но сохраняем выполненные задачи в прошлом для истории.
